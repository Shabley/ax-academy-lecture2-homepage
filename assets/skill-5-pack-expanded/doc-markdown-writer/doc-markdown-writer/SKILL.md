# Doc Markdown Writer

주제·키워드·메모를 입력받아 **실무문서**(보고서·요약·정리노트·기술문서)를 완성형 **마크다운(.md)** 파일로 생성하는 스킬이다.

블로그·뉴스레터 같은 "읽는 콘텐츠"가 아니라, **참조하고 공유하고 복사해 쓰는 문서**가 목표다. 그래서 화려한 수사보다 **명확한 구조, 스캔 가능성, 표준 마크다운 호환성**을 최우선으로 한다.

---

## 핵심 원칙

1. **결론 먼저(Top-down).** 문서 상단에 핵심 요약(TL;DR 또는 한 줄 요약)을 두고, 이후 근거·세부를 붙인다.
2. **표준 마크다운(GFM) 위주.** 어디서든 깨지지 않게 GitHub Flavored Markdown 문법만 사용한다. HTML 태그·인라인 스타일·색상 코드는 넣지 않는다(요청이 없으면).
3. **스캔 가능성.** 제목 위계(`#`/`##`/`###`)를 일관되게 쓰고, 핵심은 표·체크리스트·불릿으로 구조화한다. 한 문단이 5줄을 넘기지 않게 끊는다.
4. **즉시 사용 가능.** 추가 편집 없이 바로 붙여넣어 쓸 수 있는 완성형으로 만든다. 플레이스홀더(`[여기에 작성]`)는 정보가 정말 없을 때만, 최소한으로.
5. **분량 자동 조절.** 입력이 짧으면 간결하게, 길고 복잡하면 섹션을 늘려 구조화한다. 군더더기 없이 정리한다.

---

## 워크플로우

### 1단계 — 입력 파악
사용자가 준 것이 무엇인지 먼저 분류한다:
- **주제/키워드만** → 해당 주제의 정리문서를 새로 구성
- **두서없는 메모/대화/자료** → 구조화·재배열하여 문서화
- **기존 문서 텍스트** → 마크다운 구조로 재포맷

필요한 정보(문서 목적, 대상 독자, 원하는 분량)가 불명확하고 중요하면 **딱 한 가지만** 묻는다. 대부분은 묻지 말고 합리적으로 가정해 바로 작성하고, 가정한 내용을 문서 상단이나 응답에 짧게 명시한다.

### 2단계 — 문서 유형 선택
아래 유형 중 입력에 가장 맞는 구조를 고른다. 애매하면 **범용 정리문서** 구조를 쓴다. (구조 템플릿은 `references/templates.md` 참조)

| 유형 | 언제 | 핵심 섹션 |
|------|------|-----------|
| 보고서 | 결과·현황·분석 전달 | 요약 → 배경 → 분석/결과 → 결론·제언 |
| 요약 노트 | 자료·회의·강의 압축 | 핵심 요약 → 주요 포인트 → 상세 → 액션아이템 |
| 정리/구조화 문서 | 두서없는 정보 정돈 | 개요 → 주제별 정리(표/불릿) → 참고 |
| 기술문서/가이드 | 사용법·절차 안내 | 개요 → 사전준비 → 단계별 절차 → 트러블슈팅 |
| README형 | 프로젝트·도구 소개 | 개요 → 설치/사용 → 구성 → 참고 |

### 3단계 — 마크다운 작성
`references/templates.md`의 해당 템플릿을 골격으로 삼아 내용을 채운다. 마크다운 문법은 `references/markdown-style.md`의 규칙을 따른다.

### 4단계 — 파일 저장 및 납품
완성한 `.md`를 `/mnt/user-data/outputs/`에 저장하고 `present_files`로 사용자에게 전달한다.
- 파일명: 내용을 반영한 한글 또는 영문 슬러그(예: `2026-2분기-AI교육-운영보고.md`).
- 저장 후 한두 문장으로 무엇을 만들었는지만 간단히 안내한다. 장황한 후설명은 하지 않는다.

## 마크다운 작성 규칙 (요약)

자세한 규칙과 예시는 `references/markdown-style.md`에 있다. 핵심만:

- 문서 맨 위 `# 제목` 하나. 그 아래 **한 줄 요약** 또는 `> 핵심 요약` blockquote.
- 위계는 `#` → `##` → `###`까지만. 그 이상 깊어지면 구조를 평탄화한다.
- 비교·항목·수치는 **표**로. 순서가 있으면 번호 목록, 없으면 불릿.
- 할 일·점검 항목은 `- [ ]` 체크박스.
- 강조는 `**굵게**` 위주. 과용 금지.
- 코드·명령어·파일명은 백틱. 여러 줄이면 펜스 코드블록(언어 지정).
- 가로줄 `---`로 큰 섹션을 분리.
- 표준 GFM만 사용. HTML·색상·CSS 금지(사용자가 명시적으로 요청한 경우 제외).

## 톤과 스타일

- 한국어 기본. 영문 자료 요청 시 영문으로 전환.
- 문장은 간결하고 단정적으로. 실무자가 빠르게 읽고 바로 쓸 수 있게.
- 전문 용어는 쓰되, 필요하면 괄호로 짧게 풀어준다.
- 과장·감성적 수사 배제. 사실·구조·실행 중심.
