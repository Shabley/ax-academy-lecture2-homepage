# 마크다운 작성 규칙 (GFM)

GitHub Flavored Markdown만 사용한다. 어디에 붙여넣어도 깨지지 않는 것이 목표다.

---

## 제목 위계

- 문서당 `#`(H1)은 **하나**. 문서 제목.
- 섹션은 `##`, 하위는 `###`까지만. `####` 이하로 내려가면 구조를 다시 평탄화한다.
- 제목에는 마침표를 찍지 않는다.

**좋음**
```markdown
# 2분기 운영 보고
## 1. 현황
### 매출
```

**나쁨** (위계 건너뜀, H4 남용)
```markdown
# 보고
#### 매출
```

---

## 강조

- 핵심어는 `**굵게**`. 한 문단에 한두 개까지만.
- `*기울임*`은 보조적으로만.
- 굵게를 제목 대용으로 쓰지 않는다(제목은 `##`를 쓴다).

---

## 목록

- **순서 있음** → 번호 목록 `1. 2. 3.`
- **순서 없음** → 불릿 `-` (별표 `*` 대신 `-`로 통일)
- **할 일 / 점검** → 체크박스 `- [ ]`, 완료는 `- [x]`
- 중첩은 2단계까지. 더 깊으면 표나 별도 섹션으로.

```markdown
## 도입 체크리스트
- [ ] 계정 발급
- [x] 권한 설정
  - 관리자 권한 포함
```

---

## 표

비교·항목·수치는 표가 가장 명확하다. 헤더를 반드시 둔다.

```markdown
| 항목 | 값 | 비고 |
|------|----|------|
| A | 100 | 증가 |
| B | 80 | 유지 |
```

- 셀 안에서 줄바꿈이 필요하면 `<br>` 대신 내용을 쪼개 여러 행으로.
- 표가 너무 넓어지면(6열 초과) 구조를 나눈다.

## 코드 / 명령어 / 파일명

- 인라인: 파일명·명령·변수는 백틱 → `config.json`, `npm install`
- 블록: 여러 줄은 펜스 코드블록 + 언어 지정

````markdown
```python
print("hello")
```
````

언어 미상이면 ` ```text `.

---

## 인용 / 강조 박스

- 핵심 요약, 주의사항은 blockquote `>`.

```markdown
> **핵심 요약:** 2분기 매출은 전분기 대비 12% 증가.

> ⚠️ **주의:** 배포 전 백업 필수.
```

이모지는 신호용으로 절제해서만(⚠️ ✅ 📌 정도). 장식 남용 금지.

---

## 구분선

큰 섹션 전환에 `---`. 너무 자주 쓰지 않는다(섹션마다 한 번 정도).

---

## 링크

```markdown
[표시할 텍스트](https://example.com)
```

날것의 URL을 그대로 노출하지 말고 텍스트를 붙인다(단, 참고문헌 목록은 예외 허용).

---

## 금지 사항 (사용자가 명시 요청하지 않는 한)

- HTML 태그(`<div>`, `<span>`, `<font>` 등)
- 인라인 CSS·색상 코드(`style=`, `#1E3A5F` 등)
- 중앙정렬·폰트 지정 같은 비표준 표현
- 과도한 이모지·장식 문자

이유: 이 문서들은 GitHub, Notion, Obsidian, 블로그 에디터 등 **다양한 곳에 붙여넣어 쓰인다.** 표준 GFM만 써야 어디서든 동일하게 렌더링된다.

---

## 좋은 문서의 체크리스트

작성 후 스스로 점검:
- [ ] 맨 위에 한 줄 요약 또는 `> 핵심 요약`이 있는가
- [ ] H1이 하나뿐인가, 위계가 일관적인가
- [ ] 비교·수치가 표로 정리됐는가
- [ ] 한 문단이 5줄을 넘지 않는가
- [ ] HTML·색상·비표준 문법이 섞이지 않았는가
- [ ] 추가 편집 없이 바로 쓸 수 있는가
